## Single round vs. multiple rounds: Differences at the architectural level

- **Single round** (one Agentic Loop): A complete execution of the `query()` function - assemble context → call API → process tool call → loop until end
- **Multiple Rounds** (One Session): A session managed by the `QueryEngine` class - spanning dozens of rounds of `submitMessage()` calls, lasting several hours

`QueryEngine` (`src/QueryEngine.ts:186`) is a **session orchestrator** on top of a single-round Agentic Loop that manages much more state than just a message list:

```
QueryEngine internal state
├── mutableMessages: Message[] ← Complete conversation history, accumulated across turns
├── readFileState: FileStateCache ← Cache read file content to avoid repeated reading
├── totalUsage: NonNullableUsage ← Cumulative token consumption (input/output/cache)
├── permissionDenials: SDKPermissionDenial[] ← Permission denial record
├── discoveredSkillNames: Set<string> ← Current turn discovered skills
└── abortController: AbortController ← Session-level interrupt control
```

## Core method of QueryEngine: submitMessage()

Every time the user enters a message, the REPL or SDK calls `submitMessage()`, which performs the complete turn initialization chain:

```typescript
// src/QueryEngine.ts:211 — Simplified submitMessage process
async *submitMessage(prompt, options?): AsyncGenerator<SDKMessage> {
  // 1. Clear turn-level tracking status
  this.discoveredSkillNames.clear()
  
  // 2. Parse the model (the user may switch the model midway)
  const mainLoopModel = userSpecifiedModel
    ? parseUserSpecifiedModel(userSpecifiedModel)
    : getMainLoopModel()
  
  // 3. Dynamically assemble System Prompt (rebuild every turn)
  const { defaultSystemPrompt, userContext, systemContext } =
    await fetchSystemPromptParts({ tools, mainLoopModel, mcpClients })
  
  // 4. Wrap permission checks (track each rejection)
  const wrappedCanUseTool = async (tool, input, ...) => {
    const result = await canUseTool(tool, input, ...)
    if (result.behavior !== 'allow') {
      this.permissionDenials.push({ tool_name: tool.name, ... })
    }
    return result
  }
  
  // 5. Call the core query() function to execute agentic loop
  yield* query({
    systemPrompt, messages: this.mutableMessages,
    tools, model: mainLoopModel, ...
  })
}
```

Key design: `submitMessage()` is an `async *Generator` - it gradually yields `SDKMessage`, allowing the caller (REPL/SDK) to display the progress in real time instead of waiting for the entire turn to end.

## Session persistence: JSONL Transcript

Each conversation event is appended to the transcript file (`src/utils/sessionStorage.ts`):

### Storage path

```
~/.claude/projects/<project-hash>/<session-id>.jsonl
```

- `project-hash` is generated by `getProjectDir(originalCwd)`, sessions in the same project directory are classified into the same subdirectory
- Each record is a line of JSON (JSONL format), supporting append writing without the need to read-modify-write the entire file
- The read limit is 50MB (`MAX_TRANSCRIPT_READ_BYTES`) to prevent overly large sessions from causing OOM

### Transcript Writer

`TranscriptWriter` (`src/utils/sessionStorage.ts:1200+`) is a write queue that ensures that concurrent message appends do not overwrite each other:

```
Writing process:
  appendEntryToFile(sessionId, entry)
    ↓
  ensureCurrentSessionFile() ← Lazy initialization: the file is created only when it is first written
    ↓
  Serialize to JSON + newline
    ↓
  appendFile(path, line) ← atomic append
    ↓
  If remote persistence is configured:
    persistToRemote(sessionId, entry)
      ├── CCR v2: internalEventWriter('transcript', entry)
      └── v1 Ingress: sessionIngress.appendSessionLog(...)
```

### Session recovery link

The recovery process triggered by the `--resume` parameter (`src/main.tsx:3620+`):

```
1. Parse resume parameters:
├── UUID format → getTranscriptPathForSession(uuid)
   ├── .jsonl file path → use directly
   └── boolean → picker of the latest session
   
2. loadTranscriptFromFile(path)
   ├── Parse by JSONL line
   ├── Filter out message type records
   └── Rebuild Message[] array

3. Restore context state:
   ├── restoreCostStateForSession(sessionId) ← Restore accumulated costs
   ├── Restore agentSetting (user-selected Agent type)
   └── If --rewind-files is present, restore the file to the snapshot at the time of the specified message

4. Create QueryEngine({ initialMessages: restoredMessages })
   └── Continue the conversation from the restored message
```

## Cost Tracking: API Usage to USD

Cost tracking runs through three modules, forming a complete record→accumulation→display link:

### Record layer: Usage in API response

Each `message_delta` event carries a `usage` field (`input_tokens`, `output_tokens`, `cache_creation_input_tokens`, `cache_read_input_tokens`). `accumulateUsage()` accumulates incremental usage to the session total.

### Accumulation layer: cost-tracker.ts

```typescript
// src/cost-tracker.ts — StoredCostState data model
type StoredCostState = {
  totalCostUSD: number // Total cost in US dollars
  totalAPIDuration: number //Total API call duration (including retries)
  totalAPIDurationWithoutRetries: number // Pure inference time without retries
  totalToolDuration: number // Total tool execution time
  totalLinesAdded: number // Add the number of lines of code
  totalLinesRemoved: number //The number of lines of code deleted
  modelUsage: { [modelName: string]: ModelUsage } // Usage split by model
}
```

`addToTotalSessionCost()` calculates the cost of each API call based on model pricing, accumulated to `totalCostUSD`. `ModelUsage` by model supports counting separately after switching models in the same session.

### Persistence: retained across restarts

```typescript
// Save to project configuration at the end of each session
saveCurrentSessionCosts(sessionId)
  → projectConfig.lastCost = totalCostUSD
  → projectConfig.lastSessionId = sessionId
  → projectConfig.lastModelUsage = modelUsage
```

### Budget circuit breaker

`QueryEngineConfig.maxBudgetUsd` provides a session-level hard budget cap. In REPL, when the accumulated fees exceed $5 (`src/screens/REPL.tsx:2208`), a fee reminder dialog box pops up - this is not a hard block, but a "soft reminder".

## Model hot switching

Switching models within a session does not lose the conversation history - because `mutableMessages` is decoupled from model selection:

```
/model sonnet → setMainLoopModelOverride('claude-sonnet-4-20250514')
  ↓
The next time submitMessage() starts:
  ↓
parseUserSpecifiedModel(userSpecifiedModel)
  → Return to new model configuration
  ↓
fetchSystemPromptParts({ mainLoopModel: newModel })
  → System Prompt reassembled based on new model capabilities
  ↓
query({ model: newModel, messages: this.mutableMessages })
  → Continue the conversation with full history + new models
```

When switching models, `contextWindowTokens` and `maxOutputTokens` are also recalculated based on the new model's specifications - for example when switching from Sonnet to Opus, the context window may change from 200K to 1M.

## File snapshot and rollback

`fileHistoryMakeSnapshot()` (`src/utils/fileHistory.ts`) automatically saves the current content before each file modification by the AI. Snapshots are tied to a specific `message.id`, allowing `--rewind-files <user-message-id>` to restore exactly the state of the file at any point in the conversation - this is more granular than git (which only tracks committed content).
